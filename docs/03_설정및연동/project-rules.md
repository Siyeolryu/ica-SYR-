# Project Rules (프로젝트 규칙)

## 개발일지 작성 규칙

모든 개발 단계별로 개발일지를 작성해야 합니다.

### 작성 위치
- **폴더**: `개발일지/` 폴더에 작성
- **파일명 형식**: `YYYY-MM-DD_개발내용.md` (예: `2024-01-15_대시보드개발.md`)

### 필수 포함 내용

개발일지는 다음 섹션들을 반드시 포함해야 합니다:

#### 1. 작성시각
```markdown
## 작성시각
YYYY년 MM월 DD일
```

#### 2. 해결하고자 한 문제
```markdown
## 해결하고자 한 문제
- 이번 개발 단계에서 해결하려고 한 문제나 구현하려는 기능을 명확하게 설명
- 왜 이 작업이 필요한지 배경 설명
```

#### 3. 해결된 것
```markdown
## 해결된 것
### 1. 기능/문제 제목
- 구체적인 구현 내용
- 사용한 기술/라이브러리
- 파일 위치
- 주요 코드나 명령어

### 2. 다음 기능/문제 제목
...
```

#### 4. 해결되지 않은 것
```markdown
## 해결되지 않은 것
### 1. 문제 제목
- 왜 해결하지 못했는지
- 향후 어떻게 해결할 계획인지

### 2. 다음 문제 제목
...
```

#### 5. 향후 개발을 위한 컨텍스트
```markdown
## 향후 개발을 위한 컨텍스트
- 다음 개발자가 이어서 개발할 때 필요한 정보
- 파일 구조 설명
- 실행 방법
- 주의사항
- 다음 단계 개발 우선순위
```

### 작성 시점
- 각 개발 단계가 완료될 때마다 작성
- 중요한 기능 구현 완료 시 작성
- 문제 해결 또는 디버깅 완료 시 작성
- 하루 개발 종료 시 작성

### 작성 원칙
1. **명확성**: 누구나 이해할 수 있도록 명확하게 작성
2. **구체성**: 추상적인 설명보다는 구체적인 내용 포함
3. **완전성**: 해결된 것과 안된 것을 모두 정직하게 기록
4. **연속성**: 이전 개발일지와 연결되도록 작성
5. **재현성**: 다른 사람이 같은 작업을 재현할 수 있도록 충분한 정보 제공

### 예시 템플릿

```markdown
# 개발일지: [개발 내용 제목]

## 작성시각
YYYY년 MM월 DD일

## 해결하고자 한 문제
[문제 설명]

## 해결된 것

### 1. [기능/문제 제목]
- 구현 내용
- 파일: `경로/파일명`
- 사용 기술: [기술 스택]

### 2. [다음 기능/문제 제목]
...

## 해결되지 않은 것

### 1. [문제 제목]
- 현재 상태
- 해결하지 못한 이유
- 향후 계획

## 향후 개발을 위한 컨텍스트

### 파일 구조
```
폴더/
├── 파일1
└── 파일2
```

### 실행 방법
```bash
명령어
```

### 다음 단계 우선순위
1. [우선순위 1]
2. [우선순위 2]
...

### 참고사항
- 주의사항이나 팁
```

## 기타 프로젝트 규칙

### 코드 작성 규칙
- TypeScript 사용 시 타입을 명확하게 정의
- 함수와 변수는 의미 있는 이름 사용
- 복잡한 로직에는 주석 추가

### 커밋 규칙
- 커밋 메시지는 명확하고 구체적으로 작성
- 작은 단위로 자주 커밋
- 기능 단위로 브랜치 생성

### 문서화 규칙
- README.md는 항상 최신 상태 유지
- API 변경 시 문서 업데이트
- 환경 설정 변경 시 설치 가이드 업데이트
